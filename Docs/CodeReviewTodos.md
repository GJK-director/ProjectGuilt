### 4. 明确同名 Buff 的层数与持续时间叠加规则

- 状态：待处理
- 优先级：中
- 来源：检查 `CharacterData.AddBuff(...)`、`GetBuffStack(...)`、`CheckBuffsByTiming(...)` 时提出
- 这个问题是什么：
  - “Buff 层数叠加”和“Buff 持续时间叠加”是两个不同问题。
  - 层数叠加问的是：同名 Buff 同时存在时，最终效果是多少层。
  - 持续时间叠加问的是：新获得同名 Buff 时，要不要影响旧 Buff 的剩余回合。
  - 如果规则不提前定清楚，以后同一张卡在不同情况下可能表现不一致。
  - 例如同样是获得强壮，有时可能应该变成 4 层持续 2 回合，有时可能应该是 3 层先消失、1 层后消失。
- 当前情况：
  - 当前 `AddBuff(...)` 每次都会新增一个独立的 `BuffData`。
  - 当前不会把同名 Buff 合并成一条数据。
  - 当前 `GetBuffStack(...)` 会把所有同名 Buff 的层数加起来。
  - 当前持续时间由每一条 `BuffData` 自己单独减少。
- 示例：
  - 如果已经有 `Strength` 3 层，持续 1 回合。
  - 又获得 `Strength` 1 层，持续 2 回合。
  - 当前代码会保存为两条 Buff。
  - 当前生效层数会暂时按 4 层计算。
  - 第一次持续时间结算后，3 层那条消失，1 层那条继续保留。
- 未来需要决定：
  - 同名 Buff 是否永远分开保存。
  - 同名且持续时间相同的 Buff 是否合并。
  - 新获得同名 Buff 时，是刷新持续时间、取较长持续时间，还是保持独立持续时间。
  - UI 以后应该显示成一条合计 Buff，还是多条来源 Buff。
- 暂不实现原因：
  - 当前阶段优先读懂现有代码。
  - 这个规则会影响 Buff 显示、回合结算、卡牌效果和后续 JSON 配置设计。

### 5. 统一旧 Buff 检测和新事件系统的回合结束入口

- 状态：待处理
- 优先级：低
- 来源：检查 `BattleTurnProcessor.EndTurn(...)` 时提出
- 这个问题是什么：
  - 当前回合结束时有两套处理入口。
  - 第一套是旧 Buff 系统：`unit.CheckBuffsByTiming("TurnEnd")`。
  - 第二套是新事件系统：`BattleEventProcessor.ProcessEvent(context)`。
  - 旧 Buff 系统目前不会自动响应新事件系统的 `TurnEnd`。
  - 所以 `EndTurn(...)` 里需要先手动处理旧 Buff，再触发新事件系统。
- 当前情况：
  - 旧 Buff 系统负责处理 Buff 持续时间减少、Buff 消失等逻辑。
  - 新事件系统负责通知“回合结束”事件，目前会影响卡牌 CD 等逻辑。
  - 这两套入口现在同时存在，功能上能正常工作。
- 未来目标：
  - 后续可以考虑让 Buff 持续时间检测也接入新事件系统。
  - 这样回合结束时只需要触发一次统一事件。
  - 回合结束相关逻辑会更集中，也更容易排查问题。
- 暂不实现原因：
  - 当前旧 Buff 系统仍然承担正式持续时间处理。
  - 直接合并会影响 Buff、CD、卡牌效果等多个系统。
  - 当前阶段优先读懂代码和推进第一版战斗闭环。

### 6. 设计 Roll 点范围预览规则

- 状态：待处理
- 优先级：中
- 来源：检查 `BattleCalculator.GetFinalClashPoint(...)` 时提出
- 这个问题是什么：
  - 玩家在选择卡牌响应敌人意图前，需要知道这张卡大概能 Roll 到多少。
  - 例如显示“当前预计点数：3 - 7”，玩家才能判断自己有没有机会拼过敌人。
  - 这属于战斗 UI 和拼点策略的核心信息。
  - 如果没有预览，玩家只能盲选，战斗决策会不清楚。
- 当前情况：
  - 当前正式结算时，会先从卡牌 `minPoint` / `maxPoint` 中 Roll 一个基础点数。
  - 然后再根据 Buff 追加修正值。
  - 例如强壮会让攻击卡最终点数增加。
  - 当前这种做法比较安全，因为真实点数只在正式结算时 Roll。
  - 但当前还没有单独的“预览范围计算”方法。
- 未来需要决定：
  - 预览范围是否显示为“基础范围 + 当前 Buff 修正”。
  - 例如基础 2 - 6，强壮 +1，预览显示 3 - 7。
  - 如果有负面 Buff，最低值是否需要提前按 0 截断。
  - 一次性 Buff、条件 Buff、只影响攻击卡的 Buff，预览时是否全部计算进去。
  - 敌人意图预览是否也显示敌人可能 Roll 到的范围。
  - 预览方法是否需要和正式结算方法共用同一套修正规则，避免 UI 显示和实际结果不一致。
- 暂不实现原因：
  - 当前先记录设计问题，不急着写代码。
  - Roll 点预览会影响 UI、Buff、拼点、敌人意图多个系统。
  - 最终怎么设计需要单独讨论。

### 7. 伤害修正运算顺序规则未决定

- 状态：待处理
- 优先级：未作出决定，后续商讨后再处理
- 来源：检查 `BattleCalculator.GetFinalDamageScaled(...)` 时提出
- 这个问题是什么：
  - 伤害提升、伤害降低、易伤、减伤以后可能有不同算法。
  - 如果全部都是百分比倍率，并且统一合并成一个总倍率，先后顺序通常不影响结果。
  - 如果以后加入固定减伤、护盾、最终伤害加成、最终伤害减免等规则，先后顺序就可能影响最终扣血。
- 当前情况：
  - 当前代码把 `DamageUp`、`DamageDown`、`Vulnerable`、`DamageReduction` 全部合并进 `damageMultiplier`。
  - 当前算法相当于先算总倍率，再统一乘基础伤害。
  - 因此当前版本里易伤和减伤没有单独的先后运算顺序。
  - 这种方式比较清楚，也方便当前原型阶段计算。
- 未来需要决定：
  - 是否继续保持“统一总倍率”规则。
  - 是否增加固定减伤、护盾、最终伤害修正等新类型。
  - 如果增加新类型，需要明确它们和百分比倍率的结算顺序。
  - UI 预览和正式结算必须使用同一套顺序规则。
- 暂不实现原因：
  - 当前总倍率规则已经够用。
  - 这属于后续战斗数值规则设计，不需要现在修改代码。

### 8. 讨论防御值是否优先削减敌人攻击点数

- 状态：待处理
- 优先级：未作出决定，后续商讨后再处理
- 来源：检查 `BattleCalculator.GetFinalDefensePointScaled(...)` 和防御规则时提出
- 这个问题是什么：
  - 如果防御只是在最终伤害上做固定减法，后期遇到高倍率增伤时会比较吃亏。
  - 例如敌人攻击点数很高，身上还有大量增伤 Buff，最终伤害被放大后，固定防御值可能挡不住太多。
  - 这样会导致防御卡在数值膨胀后存在感下降。
- 当前设计倾向：
  - 更倾向让防御值优先削减敌人的攻击点数。
  - 也就是先计算：敌人修正后攻击点数 = 敌人 Roll 点数 - 防御点数。
  - 然后再用剩余攻击点数进入伤害公式和倍率计算。
- 示例：
  - 敌人 Roll 点数为 10。
  - 防御点数为 5。
  - 如果先算伤害再减防御，高倍率下可能仍然受到大量伤害。
  - 如果先削点数，则敌人只用剩余 5 点进入伤害计算。
- 未来需要决定：
  - 防御点数是否能把敌人攻击点数削到 0。
  - 削到 0 时是否完全无伤。
  - 防御成功和防御未完全成功是否还需要不同表现。
  - 防御点数是否参与 UI 预览。
  - 敌人增伤 Buff 是作用在削点前，还是削点后。
  - 这个规则如何接入第一版防御 / 闪避 / 无人响应结算。
- 暂不实现原因：
  - 当前只讨论设计思路，不修改代码。
  - 防御规则会影响战斗平衡、UI 预览、伤害公式和后续 Buff 设计。

### 10. 将执行计划执行器接回正式战斗结算

- 状态：待处理
- 优先级：中
- 来源：检查 `BattleExecutionPlanExecutor` 和 `BattleResolver` 时提出
- 这个问题是什么：
  - 当前 `BattleExecutionPlanExecutor` 为了先跑通 Action Slot 执行计划，写了一套简化执行逻辑。
  - 它会直接 Roll 点、比较点数、调用 `TakeDamage(...)`、标记执行项完成。
  - 但它没有走 `BattleResolver` 的完整结算流程。
  - 如果长期保留两套结算路径，后续容易出现同一张卡在不同入口下表现不一致。
- 当前情况：
  - `BattleResolver` 更接近当前已有的正式战斗结算器。
  - `BattleResolver` 已经处理拼点、伤害公式、事件链、卡牌效果、CD / UseCount / 负罪感等逻辑。
  - `BattleExecutionPlanExecutor` 当前只处理第一版简化逻辑。
  - 当前 `Executor` 不触发完整事件链，不处理完整 Buff 修正，不处理完整 CD / UseCount / guiltGain。
- 未来目标：
  - `BattleExecutionPlanExecutor` 只负责按执行计划顺序分派任务。
  - `BattleResolver` 或未来正式结算入口负责真正拼点、伤害、事件、卡牌效果、CD、UseCount、负罪感。
  - 也就是：`Executor = 调度员`，`Resolver = 结算员`。
  - 后续需要给 `BattleResolver` 增加更适合执行计划调用的公开入口，而不是继续依赖测试函数名。
- 暂不实现原因：
  - 当前还在读懂代码和梳理职责阶段。
  - 直接合并会影响执行计划、敌人意图、防御 / 闪避、无人响应、卡牌使用状态等多个系统。

### 11. 设计基于槽位、敌人意图和速度的执行顺序规则

- 状态：待处理
- 优先级：偏高
- 来源：检查 `BattleExecutionPlanManager.CreateBasicExecutionPlan(...)` 时提出
- 这个问题是什么：
  - 当前执行计划生成逻辑只简单区分“已响应敌人意图”和“无人响应敌人意图”。
  - 它还没有真正处理战斗开始后谁先行动、哪个槽位先触发、敌人速度是否抢先等规则。
  - 这会直接影响第一版战斗闭环的核心体验。
- 当前情况：
  - 当前默认是先加入已响应敌人意图，再加入无人响应敌人意图。
  - 当前没有按玩家槽位 1 / 槽位 2 排序。
  - 当前没有按敌人速度和玩家速度重新排序。
  - 当前没有处理“敌人攻击某个槽位时，该槽位是否被提前触发”。
- 需要讨论的规则：
  - 默认情况下，玩家槽位是否按槽位 1 -> 槽位 2 执行。
  - 如果敌人速度大于玩家，并且敌人攻击玩家槽位 2，是否应该让槽位 2 先触发。
  - 如果玩家速度大于敌人，玩家是否可以按自己的槽位顺序先处理。
  - 多个敌人意图攻击不同角色 / 不同槽位时，如何决定先后。
  - 已响应敌人意图、无人响应敌人意图、自由行动三者如何混排。
  - 同一个槽位既被敌人攻击，又安排了自由行动时，应该先处理哪一边。
- 未来目标：
  - 在执行计划生成阶段明确处理这些顺序规则。
  - `BattleExecutionPlanManager` 或未来新的 `BuildExecutionPlan` 方法负责生成真正执行顺序。
  - `BattleExecutionPlanExecutor` 只按生成好的顺序分派，不在执行时临时决定顺序。
- 暂不实现原因：
  - 当前还在读懂代码和整理设计规则。
  - 这个规则需要先和战斗策划一起明确，否则容易写出错误执行顺序。
